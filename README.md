# Yandex Reviews Parser

Небольшое SPA-приложение: авторизация через Sanctum, ввод ссылки на организацию в Яндекс.Картах, парсинг отзывов и постраничный вывод.

**Стек:** Laravel 12 (API) + Vue 3 SPA (Composition API) + MySQL + Laravel Sanctum.

---

## Быстрый старт

### Вариант 1 — Docker (рекомендуется)

Нужен только [Docker](https://docs.docker.com/get-docker/).

```bash
docker-compose up --build
```

Поднимается два контейнера: **app** (PHP + Nginx) и **db** (MySQL 8.4). Приложение доступно на [http://localhost:8000](http://localhost:8000).

При первом запуске entrypoint автоматически:
- создаёт `.env` из `.env.example`;
- генерирует `APP_KEY`;
- запускает миграции (ждёт готовности MySQL через healthcheck);
- засевает тестового пользователя.

### Вариант 2 — локально (PHP + Node)

**Требования:** PHP 8.2+, Composer, Node.js 20+.

```bash
# 1. Зависимости
composer install
npm install

# 2. Окружение
cp .env.example .env
php artisan key:generate

# 3. База данных (MySQL должен быть запущен локально)
php artisan migrate
php artisan db:seed

# 4. Фронт
npm run build
# или для dev-режима с HMR:
# npm run dev

# 5. Сервер
php artisan serve
```

Открыть [http://localhost:8000](http://localhost:8000).

## Подход к парсингу

### Почему не headless-браузер

Яндекс.Карты при каждом открытии карточки организации вставляют в HTML страницы JSON-блок с начальным состоянием приложения (`{"config": ...}`). В этом блоке есть `csrfToken`, `requestId`, `locale`, `experiments` и ID организации — всё необходимое для запросов к внутреннему API.

Headless-браузер (Puppeteer / Playwright) решил бы задачу, но добавил бы ~150 МБ зависимостей и значительный оверхед по памяти/CPU на каждый парсинг. Вместо этого используется прямой HTTP-парсинг:

1. **Первый запрос** — `GET` карточки с обычным User-Agent браузера. Яндекс отдаёт HTML с JSON в теге `<script>`. Парсим его регуляркой и извлекаем `config`.

2. **Пагинация отзывов** — внутренний endpoint `https://yandex.ru/maps/api/business/fetchReviews` принимает те же параметры, что передаёт клиентский JS при прокрутке. Запросы идут постранично (по 50) в цикле, пока есть страницы или не достигнут лимит (`YANDEX_MAX_REVIEWS`).

3. **CSRF-ротация** — если Яндекс возвращает обновлённый `csrfToken` в ответе, он подхватывается и используется для следующего запроса.

4. **Сессионные cookie** — `CookieJar` (Guzzle) сохраняет cookie между запросами в рамках одного парсинга, что снижает вероятность блокировки.

5. **Подпись запроса** — endpoint требует параметр `s` (числовой хэш djb2 от отсортированной query string). Это та же логика, что в клиентском JS Яндекса.

### Ограничения

- Яндекс периодически меняет структуру JSON. Если `extractInitialState` не найдёт нужный блок, парсер выбросит `YandexMapsParserException` с понятным сообщением.
- При агрессивном парсинге Яндекс может показать CAPTCHA — это тоже обрабатывается и сообщается пользователю.

## Кэширование отзывов

Все отзывы сохраняются в БД (`organization_reviews`) при каждом сабмите ссылки. Страница отзывов работает с уже сохранёнными данными — повторных обращений к Яндексу не происходит. Такой подход оправдан потому что:

- парсинг ~600 отзывов занимает несколько секунд — делать это при каждой смене страницы было бы неприемлемо;
- пользователь всегда может нажать «Сохранить» ещё раз, чтобы обновить данные.

## Что доделал бы при наличии времени

- **Несколько организаций** — сейчас показывается только последняя; добавил бы список с возможностью переключения.
- **Планировщик обновлений** — автоматический ре-парсинг раз в N часов через `artisan schedule`.
- **Прокси-ротация** — для обхода блокировок при частом парсинге.
- **Тесты** — unit-тест для `YandexMapsParser` (с мок-HTML) и feature-тест для API.
- **Vue Router** — разделить логин, настройки и список отзывов на отдельные маршруты.
